Проект: GUtv Booker Event

1. Что это за проект

Это веб-приложение для подачи и обработки заявок на съемку или сопровождение мероприятий.
Сервис нужен для того, чтобы представители организаций и администраторы могли:

- создавать заявки на мероприятия;
- указывать причину, дату и время проведения;
- отслеживать статус заявки в личном кабинете;
- просматривать комментарии по заявке;
- администрировать пользователей и управлять их ролями.

По текущей структуре проекта сервис работает как единая система:

- публичная часть сайта с описанием сервиса;
- регистрация и вход пользователей;
- личный кабинет;
- административная панель;
- API для работы с заявками, пользователями и авторизацией.

2. Для чего нужен проект

Основная задача проекта — перевести процесс подачи event-заявок в цифровой вид.
Это позволяет:

- убрать ручной сбор заявок через сообщения и таблицы;
- хранить заявки централизованно;
- ускорить проверку и согласование;
- видеть историю заявок и их статусы;
- разграничить доступ между обычными пользователями и администраторами.

3. Технически что используется

Проект реализован на следующем стеке:

- Next.js 16 + React 19;
- TypeScript;
- Prisma ORM;
- SQLite как база данных;
- JWT-авторизация;
- Node.js runtime.

Важно: сейчас проект использует SQLite, то есть база данных хранится в одном локальном файле.
Это удобно для небольшого количества пользователей и простой установки, но накладывает ограничения:

- приложение лучше запускать в одном экземпляре;
- обязательно нужно хранить файл базы на постоянном диске;
- нужно делать резервные копии файла базы данных;
- для роста нагрузки в будущем лучше будет перейти на PostgreSQL.

4. Что нужно для хостинга на сайте университета

Проекту нужен серверный хостинг, а не только обычное размещение статических HTML-файлов.
Причина: здесь есть backend, авторизация, API и база данных.

Поэтому размещение должно поддерживать:

- запуск Node.js-приложения;
- постоянный фоновый процесс;
- хранение файла базы данных;
- настройку переменных окружения;
- reverse proxy через Nginx или Apache;
- HTTPS-сертификат.

5. Рекомендуемый вариант размещения

Оптимальный вариант для университета:

- виртуальная машина Linux;
- запуск приложения в Docker-контейнере;
- отдельный volume или папка на сервере для SQLite-базы;
- Nginx перед приложением как reverse proxy.

Почему Docker удобен:

- проще переносить проект между серверами;
- проще фиксировать окружение и версию Node.js;
- проще перезапускать и обновлять приложение;
- меньше риск "на одном сервере работает, на другом нет".

Если Docker в университете использовать нельзя, проект можно запускать напрямую на виртуальной машине как обычный Node.js-сервис через systemd.

6. Минимальные и рекомендуемые ресурсы

Для текущей версии проекта без большой нагрузки:

Минимально:

- 1 vCPU;
- 2 GB RAM;
- 10 GB SSD;
- Linux Ubuntu 22.04/24.04 или аналогичный серверный дистрибутив;
- Node.js 20+;

Рекомендуемо:

- 2 vCPU;
- 4 GB RAM;
- 20 GB SSD;
- Docker + Docker Compose;
- Nginx;
- автоматический перезапуск сервиса;
- резервное копирование базы данных.

Если пользователей немного и одновременно работает мало людей, 2 GB RAM хватит.
Если планируется постоянная работа админки, несколько одновременных пользователей и развитие проекта, лучше сразу закладывать 4 GB RAM.

7. Что должно быть доступно на сервере

Для запуска проекта на сервере нужны:

- Node.js 20 или новее;
- npm;
- доступ на запись в папку, где лежит SQLite-база;
- переменные окружения;
- открытый порт для приложения внутри сервера (обычно 3000);
- внешний доступ через Nginx/Apache на 80/443 портах.

8. Переменные окружения

Минимально проекту нужны такие переменные окружения:

- DATABASE_URL
- JWT_SECRET
- JWT_ISSUER
- JWT_AUDIENCE
- JWT_EXPIRE_MINUTES

Пример логики:

- DATABASE_URL указывает путь к SQLite-файлу;
- JWT_SECRET хранит секрет для токенов;
- JWT_ISSUER и JWT_AUDIENCE задают параметры JWT;
- JWT_EXPIRE_MINUTES задает время жизни access token.

9. Что важно учесть при размещении именно у университета

Если проект будет доступен как часть сайта университета, желательно заранее согласовать:

- будет ли это отдельный поддомен, например event.university.ru;
- можно ли использовать Docker;
- можно ли запускать постоянный Node.js-процесс;
- где будет храниться база данных;
- как будут делаться бэкапы;
- кто будет отвечать за обновления и перезапуск.

На практике самый удобный вариант — отдельный поддомен и отдельная виртуальная машина или контейнер.

10. Краткий вывод для администраторов хостинга

Проект нельзя разместить как обычный статический сайт.
Для него нужен сервер с поддержкой Node.js-приложения и хранением данных.

Предпочтительный вариант:

- Linux VM;
- Docker-контейнер с приложением;
- Nginx как reverse proxy;
- 2 vCPU, 4 GB RAM, 20 GB SSD;
- постоянное хранилище для SQLite;
- HTTPS и резервное копирование.

Если нужен самый простой рабочий вариант, можно запустить и без Docker:

- Linux VM;
- Node.js 20+;
- systemd service;
- Nginx;
- 1 vCPU, 2 GB RAM, 10 GB SSD минимум.

11. Замечание на будущее

Если сервисом начнут активно пользоваться, желательно перейти с SQLite на PostgreSQL.
Для начального университетского размещения текущая архитектура подойдет, но для роста и более надежной многопользовательской работы PostgreSQL будет правильнее.
